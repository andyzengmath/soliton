# Risk Scoring Factors

**Invariant:** Factor weights must sum to 100%. Current total: 25 + 20 + 20 + 15 + 10 + 10 = 100%.

## Factor Definitions

### 1. Blast Radius (Weight: 25%)

How many other files depend on the changed files.

**Scoring methodology:** Use Grep to count files that import or reference each changed file across the codebase. Score = min(100, totalImporterCount * 10).

| Score Range | Example |
|-------------|---------|
| 0-20 (LOW) | Leaf file with 0-2 importers (test helper, config) |
| 30-50 (MEDIUM) | Utility used by 3-5 files |
| 60-80 (HIGH) | Core module imported by 6-8 files |
| 90-100 (CRITICAL) | Shared library imported by 10+ files |

### 2. Change Complexity (Weight: 20%)

Ratio of control-flow changes vs cosmetic changes.

**Scoring methodology:** Analyze the diff for control-flow additions/modifications (if/else, try/catch, loops, switch, return statements, async/await, error handling) vs cosmetic changes (renames, formatting, comments, whitespace). Score = controlFlowLines / totalAddedLines * 100.

| Score Range | Example |
|-------------|---------|
| 0-20 (LOW) | Variable rename, comment update, formatting fix |
| 30-50 (MEDIUM) | New function with simple logic, config change |
| 60-80 (HIGH) | New error handling paths, conditional branching |
| 90-100 (CRITICAL) | Complex state machine, concurrent logic, retry/backoff |

### 3. Sensitive Paths (Weight: 20%)

Whether changed files match sensitive path patterns.

**Scoring methodology:** Check each changed file path against patterns from rules/sensitive-paths.md and user-configured patterns. Score = 100 if any changed file matches a sensitive pattern, 0 if none match.

| Score Range | Example |
|-------------|---------|
| 0 (NONE) | README.md, docs/, tests/ |
| 100 (MATCH) | auth/middleware.js, payments/stripe.py, .env.production |

### 4. File Size and Scope (Weight: 15%)

Total volume of changes across files.

**Scoring methodology:** Based on total lines changed (additions + deletions) and number of files.

| Lines Changed | Score |
|---------------|-------|
| < 50 | 10 |
| 50-200 | 30 |
| 200-500 | 60 |
| 500-1000 | 80 |
| > 1000 | 100 |

### 5. AI-Authored Signals (Weight: 10%)

Indicators that code may be AI-generated.

**Scoring methodology:** Look for these signals in the diff and git log:
- Unusually uniform formatting and style within new code
- High boilerplate ratio (repetitive patterns, template-like code)
- Agent signatures in commit messages (e.g., "Co-Authored-By: Claude", "Generated by Copilot")
- Large blocks of new code with no incremental commits

Score = weighted sum of detected signals, 0-100.

| Score Range | Example |
|-------------|---------|
| 0-20 (LOW) | Human-looking incremental changes, varied style |
| 30-60 (MEDIUM) | Some uniform patterns, partial agent signatures |
| 70-100 (HIGH) | Clear agent signatures, large uniform blocks, boilerplate-heavy |

### 6. Test Coverage Gap (Weight: 10%)

Production files changed without corresponding test changes.

**Scoring methodology:** Identify production files (non-test, non-config) in the changeset. Check if corresponding test files were also modified. Score = (production files without test changes / total production files) * 100.

| Score Range | Example |
|-------------|---------|
| 0 (NONE) | All production changes have corresponding test changes |
| 50 (PARTIAL) | Half of changed files lack test updates |
| 100 (FULL GAP) | No test files changed despite production changes |
