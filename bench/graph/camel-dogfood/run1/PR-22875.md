:yellow_circle: [correctness] Awaitility replacement drops MockEndpoint expectation semantics in `components/camel-atom/src/test/java/org/apache/camel/component/atom/AtomGoodBlogDefaultIdempotencyTest.java`:88 (confidence: 85)
The original test used `mock.expectedMessageCount(7)` + `mock.assertIsSatisfied()`. `assertIsSatisfied()` validates all registered Camel mock expectations using an internal CountDownLatch that releases as soon as conditions are met. Replacement uses `assertEquals(7, mock.getReceivedCounter())` polled by Awaitility — strictly weaker. Two problems: (1) if the atom polling consumer delivers an 8th message (idempotent filter race), `getReceivedCounter()` becomes 8, `assertEquals` throws permanently, Awaitility times out — whereas `assertIsSatisfied()` would have reported the discrepancy immediately; (2) any mock expectations registered in superclass / `@BeforeEach` are silently skipped. The two sibling files correctly retain `expectedMessageCount + assertIsSatisfied(10000)` — the first file's rewrite is inconsistent.
```suggestion
mock.expectedMessageCount(7);
await().atMost(10, TimeUnit.SECONDS)
       .untilAsserted(() -> mock.assertIsSatisfied());
```
