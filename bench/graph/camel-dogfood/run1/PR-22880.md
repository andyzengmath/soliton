## Critical

:red_circle: [correctness] trustManagerMapper returns lambda that can produce null TrustManager array, causing NPE during SSL handshake in `components/camel-vertx/camel-vertx-common/src/main/java/org/apache/camel/component/vertx/common/TrustManagerFactoryOptions.java`:47 (confidence: 88)
Null guard added in this PR only checks `trustManagerFactory == null` — does NOT guard against `trustManagerFactory.getTrustManagers()` returning null or empty array. The returned lambda `serverName -> trustManagerFactory.getTrustManagers()` is invoked by Vert.x's SSLHelper during the SSL handshake. If `getTrustManagers()` returns null (factory constructed but `init()` never called, or initialized with empty material), Vert.x throws NPE inside the SSL pipeline — a cryptic failure at connection time rather than a clear configuration error at startup. Directly asymmetric with the `KeyManagerFactoryOptions` guard added in the same PR which checks both `getKeyManagers() == null` AND `length == 0`. The trust path is the more security-sensitive one.
```suggestion
if (trustManagerFactory == null || trustManagerFactory.getTrustManagers() == null
        || trustManagerFactory.getTrustManagers().length == 0) {
    return null;
}
```

## Improvements

:yellow_circle: [security] Silent fallback to JVM default trust store when truststore unconfigured may mask server-side mTLS misconfiguration in `components/camel-vertx/camel-vertx-common/src/main/java/org/apache/camel/component/vertx/common/VertxHelper.java`:69 (confidence: 75)
When `trustManagerFactory == null` and SSL is enabled, `setupSSLOptions` skips `setTrustOptions()` entirely — Vert.x falls back to JVM default trust store. For CLIENT use this is acceptable. For SERVER expecting mTLS, this creates a silent-degradation hazard: a server that forgets `trustManagers` will silently start without client certificate verification instead of failing fast. Pre-PR behavior was likely fail-closed via NPE during handshake; new behavior is fail-open. Combined with the absence of a negative test for this misconfig path, users get a working-but-insecure deployment.

:yellow_circle: [correctness] Both factories null with SSL enabled silently falls back to JDK default trust store instead of failing fast (confidence: 78)
After this PR, when both `createKeyManagerFactory()` and `createTrustManagerFactory()` return null, `setupSSLOptions` still calls `setSsl(true)` but skips both `setKeyCertOptions` and `setTrustOptions`. Vert.x uses the JDK default SSLContext. Behavioral change: pre-PR code would have caused immediate NPE/SSL init error, making misconfiguration visible. New behavior allows silent ignore of intended `SSLContextParameters`.

:yellow_circle: [security] Test coverage gap — no mTLS test, no negative test for missing-truststore-on-server scenario (confidence: 80)
Tests cover only one-way SSL happy path. No mTLS test. No negative test for the dangerous misconfiguration the PR explicitly relaxes. The new failure modes are precisely the ones not covered.
