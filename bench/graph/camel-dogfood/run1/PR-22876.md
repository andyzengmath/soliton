## Critical

:red_circle: [correctness] `Files.exists` follows symlinks — dangling symlink from partial run causes FileAlreadyExistsException in `dsl/camel-jbang/camel-jbang-core/src/main/java/org/apache/camel/dsl/jbang/core/commands/ExportBaseCommand.java`:1286 (confidence: 88)
`Files.exists(linkInExportDir)` uses default `LinkOption` which follows symbolic links. If a previous partial run left a dangling symlink (link exists, target gone), `Files.exists` returns false because the link target is missing → `Files.delete` is skipped → `Files.createSymbolicLink` throws `FileAlreadyExistsException` because the symlink inode is still present. Reproducible whenever an export is interrupted after `PathUtils.copyDirectory` but before `PathUtils.deleteDirectory`, or when `exportDir` already contains a stale symlink from a prior dev-mode run.
```suggestion
// Replace Files.exists check with Files.deleteIfExists which operates on the link inode:
Files.deleteIfExists(linkInExportDir);
```

## Improvements

:yellow_circle: [correctness] `copyLocalLibDependencies` lost symlink behavior — not added to deferredSymlinks (confidence: 82)
Pre-PR, `copyLocalLibDependencies` called `ExportHelper.safeCopy(sourcePath, targetPath, true, symbolicLink)` which created a symlink in-place when `symbolicLink=true`. PR removed `symbolicLink` parameter from `safeCopy` but did NOT add a `deferredSymlinks.put(targetPath, sourcePath)` entry. Same omission in `copyApplicationPropertiesFiles`. Result: lib JARs and profile-specific properties are always copied as regular files even in dev-mode, defeating the live-reload intent.

:yellow_circle: [correctness] `relativize` between mismatched absolute/relative Paths throws IllegalArgumentException on Windows (confidence: 75)
`buildDir = Path.of(BUILD_DIR)` is relative; if any caller absolutizes intermediate paths, `relativize` throws `IllegalArgumentException: 'other' is different type of Path`. Cross-drive relativize on Windows also throws. Fix: absolutize both sides before relativizing.

:yellow_circle: [correctness] Deprecated `safeCopy(File, File, boolean)` silently drops symlink behavior for subclass callers (confidence: 70)
Pre-PR forwarded `symbolicLink` to `ExportHelper.safeCopy`; post-PR doesn't add to `deferredSymlinks`. Subclasses (or legacy code) calling this `@Deprecated` method when `symbolicLink=true` silently produce a plain copy instead of a symlink — backward compatibility broken.
