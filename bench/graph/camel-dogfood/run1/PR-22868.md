Only `camel-base` (core) `IOConverter.toObjectInput`/`toObjectOutput` registrations are removed. `camel-mina` and `camel-netty` still ship their own `@Converter`-annotated `toObjectInput(...)` methods (`MinaConverter.java:71`, `NettyConverter.java:91`) that wrap raw `ObjectInputStream` over `IoBuffer` and `ByteBuf`. So `getBody(ObjectInput.class)` against a Mina/Netty consumer message will still trigger Java deserialization on attacker-controlled wire bytes. Those component converters install an `ObjectInputFilter` (default `!java.net.**;java.**;javax.**;org.apache.camel.**;!*`) — meaningful defense, but gadget chains using only `java.**` classes (e.g., `BadAttributeValueExpException`-style) remain reachable. The upgrade-guide should scope the claim to camel-base/core and cross-link to "Default deserialization filter tightened".
