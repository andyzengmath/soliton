## Improvements

:yellow_circle: [correctness, cross-file-impact] Public-API immutability contract regression — `Collectors.toList()` returns a mutable list; previous code returned an unmodifiable view in `src/main/java/org/springframework/samples/petclinic/vet/Vet.java`:60 (confidence: 92)
The pre-refactor `getSpecialties()` returned `Collections.unmodifiableList(sortedSpecs)`, which threw `UnsupportedOperationException` on any `add` / `remove` / `set` / `clear` call. The post-refactor `Collectors.toList()` makes no such guarantee — per the [JDK 21 javadoc](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/stream/Collectors.html#toList()): "There are no guarantees on the type, mutability, serializability, or thread-safety of the List returned." Current OpenJDK implementations return a plain `ArrayList`, so any caller that previously got `UnsupportedOperationException` on mutation now silently succeeds and corrupts a derived list (the per-call returned snapshot, not the underlying `Set<Specialty>`, but a consumer that mutates expecting protection now goes unprotected). This is a `@XmlElement`-annotated public-API method on a JPA entity bean — the immutability invariant is part of its observable contract for JAXB consumers and any future caller. (Mitigated by: graph CALLS edges + Grep of `src/` confirm no internal caller currently mutates the returned list — `ClinicServiceTests.java:211-212` does read-only `.get(0).getName()`, `vetList.html:20` does a read-only Thymeleaf iteration. Severity downgraded from CRITICAL to IMPROVEMENT given zero current runtime impact, but a future caller that relies on the documented unmodifiable contract will silently misbehave, and any external JAXB-marshalled consumer no longer gets the runtime guard.)
```suggestion
	@XmlElement
	public List<Specialty> getSpecialties() {
		return getSpecialtiesInternal().stream()
			.sorted(Comparator.comparing(NamedEntity::getName))
			.toList();
	}
```
References: https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/stream/Collectors.html#toList(), https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/stream/Stream.html#toList()

:yellow_circle: [hallucination] Non-idiomatic stream terminal op for Java 17 — prefer `Stream.toList()` over `Collectors.toList()` in `src/main/java/org/springframework/samples/petclinic/vet/Vet.java`:63 (confidence: 88)
The project targets Java 17 (`<java.version>17</java.version>` in `pom.xml`). Since Java 16, `Stream.toList()` is the preferred terminal operation: it is more concise, allocates one less collector object, and — critically here — returns an UNMODIFIABLE list that preserves the previous `Collections.unmodifiableList(...)` contract. Adopting it removes the need for the `import java.util.stream.Collectors` line as well, shrinking the diff by one line. This single change resolves the immutability regression in the previous finding, so the two issues should be fixed together.
```suggestion
	@XmlElement
	public List<Specialty> getSpecialties() {
		return getSpecialtiesInternal().stream()
			.sorted(Comparator.comparing(NamedEntity::getName))
			.toList();
	}
```
