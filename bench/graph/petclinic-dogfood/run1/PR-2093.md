## Findings

### CRITICAL — Build supply-chain integrity regression: `distributionSha256Sum` removed from gradle wrapper

```
agent: security
category: security
severity: critical
confidence: 92
file: gradle/wrapper/gradle-wrapper.properties
lineStart: 3
lineEnd: 4
title: distributionSha256Sum line deleted; gradle wrapper download is no longer integrity-verified
references: [CWE-494 (Download of Code Without Integrity Check), OWASP A08:2021 Software & Data Integrity Failures, https://docs.gradle.org/current/userguide/gradle_wrapper.html#sec:verification]
```

**Pre-PR:**
```properties
distributionSha256Sum=ed1a8d686605fd7c23bdf62c7fc7add1c5b23b2bbc3721e661934ef4a4911d7c
distributionUrl=https\://services.gradle.org/distributions/gradle-8.14.3-all.zip
```

**Post-PR:**
```properties
distributionUrl=https\://services.gradle.org/distributions/gradle-8.14.3-bin.zip
```

**Description.** The PR removes the pinned `distributionSha256Sum` while simultaneously changing the Gradle distribution URL from `-all.zip` to `-bin.zip`. The hash was specific to the `-all.zip` artifact, so a naïve URL swap without re-pinning would have failed verification — but instead of recomputing the hash for the new artifact, the hash was deleted entirely. This is a real attack surface: any attacker who can MITM `services.gradle.org` (or an MVNW_REPOURL mirror, or a corp proxy that re-signs TLS) can ship arbitrary code into every build, including CI. `validateDistributionUrl=true` is **NOT** a substitute — that flag only checks that the URL host matches a `gradle.org` pattern, not that the bytes are intact.

The kept `mvnw` script (lines 227-234 of the post-PR file) still has matching SHA-256 enforcement code, which means Maven's wrapper is verifiable but Gradle's is not — asymmetric supply-chain posture in the same repo.

**Suggestion.** Re-pin the hash for the `-bin.zip` variant. The official checksum is published at `https://services.gradle.org/distributions/gradle-8.14.3-bin.zip.sha256`. Add back:

```properties
distributionSha256Sum=<sha256 of gradle-8.14.3-bin.zip published by gradle.org>
```

Equivalently run `./gradlew wrapper --gradle-version 8.14.3 --distribution-type bin` against a known-good Gradle and commit the regenerated file — that flow writes the hash automatically.

**Why this matters in this codebase.** PetClinic is canonical Spring reference material, mirrored and forked widely. A poisoned wrapper in upstream propagates into every fork's CI. The cost of restoring the line is a single `sha256` lookup; the cost of the regression is build integrity for the entire downstream graph.

### IMPROVEMENT — Postgres 18.0 in k8s/db.yml without migration note

```
agent: correctness
severity: improvement
confidence: 70
file: k8s/db.yml, docker-compose.yml, README.md
title: postgres:17 → 18 is a MAJOR DB version; k8s manifest will silently roll forward on next pod replace
```

PostgreSQL 18 changed several defaults (e.g., default `wal_level`, async I/O, scram-sha-256 enforcement nuances). For PetClinic itself this is fine — it's a demo with ephemeral state. For the **enterprise dogfood signal**: any team that copy-pastes `k8s/db.yml` as a starter and has a stateful PVC attached will trigger a major-version upgrade on the next pod replace, which is unsafe without `pg_upgrade` or dump/restore.

**Suggestion.** Add a comment block in `k8s/db.yml` above the image line: `# postgres:18.0 — MAJOR. Schema-incompatible with 17.x volumes; use pg_upgrade for stateful migrations.`

## Realist Check (post-synthesis pass on the CRITICAL)

Pressure-test on the gradle-sha256 finding:

1. **Reachability:** YES. Every `./gradlew` invocation triggers wrapper download on a fresh clone or after wrapper cache eviction; CI runners always hit this path.
2. **Exploitability:** Realistic. Threat models include: corporate MITM proxies that re-sign HTTPS (real in some enterprise deployments), DNS hijack of `services.gradle.org`, BGP-level rerouting, cache-poisoning of an `MVNW_REPOURL` mirror. None are exotic — they are the textbook reasons the integrity-pin field exists in the first place.
3. **Blast radius:** Wide. Any consumer of this repo (forks, downstream CI, tutorial readers copy-pasting). PetClinic is a teaching artifact — `validateDistributionUrl=true` is decorative.
4. **Mitigating factors searched:** None present in the repo. `validateDistributionUrl=true` only checks URL hostname pattern, not bytes. There is no organization-level pinning, no Gradle Enterprise checksum verification configured, no separate `gradle-wrapper.jar` hash verification step in CI.

Decision: **confirmed_critical**. No mitigation citation possible. Severity HELD at CRITICAL.

> Note on judgment calibration: this is the kind of finding an LLM swarm easily under-weights — there's no broken code, the build passes, and the PR is human-reviewed and approved by a project maintainer. The error is one of *omission* relative to the prior secured state, which is exactly the failure mode integrity-pin fields were designed to prevent. The realist-check pass keeps it CRITICAL because the standard for downgrading is a cited, file-line mitigation, and no such mitigation exists in this repo.
