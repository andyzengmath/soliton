## Findings

### IMPROVEMENT — `${status}` is a real Spring binding (no hallucination)

Verified that Spring Boot's `DefaultErrorAttributes` populates `status` (Integer) on the error model — corroborated locally by `src/test/java/.../system/CrashControllerIntegrationTests.triggerExceptionJson` which asserts `getBody().containsKey("status")`. The new `${status}` reference in `error.html` is correctly bound. The `${message}` reference (already present pre-PR) is the same `DefaultErrorAttributes` field, gated by `spring.web.error.include-message=ALWAYS` in test config; production behavior is unchanged from before this PR.

### IMPROVEMENT — `th:case="404"` numeric-vs-string semantics

`${status}` is an `Integer` (per `DefaultErrorAttributes`). `th:case="404"` is parsed by Thymeleaf's Standard Expression engine as a numeric literal, so the comparison is `Integer.equals(Integer)` and works. Verified consistent with Thymeleaf 3.x semantics. **No fix required**, but if the reviewer wants belt-and-suspenders robustness across Thymeleaf upgrades, `th:case="'404'"` (quoted string) plus letting Thymeleaf's `#strings.toString(status)` coerce works equally well in current versions. This is a stylistic preference, not a bug.

### IMPROVEMENT — Wildcard fallback handles null `${status}`

If `${status}` were ever null (e.g. unusual error path before Spring populates the model), `th:switch` would not match `404` or `500`, and `th:case="*"` correctly catches null. The user sees `error.general` text — the desired graceful-fallback behavior. No null guard needed.

### IMPROVEMENT — Coverage is 404/500 only; other status codes fall through

403 (Forbidden), 401 (Unauthorized), 400 (Bad Request), 503 (Service Unavailable), and any other non-404/non-500 status will display `error.general` ("An unexpected error occurred."). Acceptable as a generic catchall and consistent with the PR's stated scope ("404, 500, other unexpected errors"). Optional follow-up if richer UX is desired — not blocking.

### IMPROVEMENT — Non-blocking nit: HTML comments

The new `<!-- Title: Something happened -->` and `<!-- Exception message (for developers) -->` comments are gratuitous (the surrounding markup is self-explanatory). The `<!-- Status-specific error message -->` comment is fine as section signage. Leaving as-is is acceptable; matches the existing template style of "no comments". Nit-level only.

### Not flagged: translation accuracy

Soliton does not evaluate translation correctness for de/es/fa/ko/pt/ru/tr. The 7 locale × 3 key submissions (21 translated strings) are accepted as-submitted, consistent with PR #1878's policy. Native-speaker review remains a maintainer responsibility.

### Not flagged: integration-test fragility

The existing `CrashControllerIntegrationTests.triggerExceptionHtml` uses `containsSubsequence("<body>", "<h2>", "Something happened...", "</h2>", "<p>", "Expected:", ..., "</p>", "</body>")`. Subsequence matching is non-contiguous, so inserting a new `<p th:switch>` between `</h2>` and the existing exception `<p>` does NOT break the assertion — it still finds `</h2>` then any `<p>` then `Expected:` further along, in order. **Not a regression.**

PR delivers what it promises: 3 message keys × 8 locales (with `messages_en.properties` correctly left as a fallback comment) plus a Thymeleaf `th:switch` block on `${status}` in `error.html`. The Spring binding (`${status}` from `DefaultErrorAttributes`) is verified against the existing CrashController integration test that asserts the same key in JSON output. The wildcard `th:case="*"` correctly handles null and any non-404/non-500 status. No correctness bugs, no hallucinated bindings, no integration-test breakage.

Translation quality is out of Soliton's evaluable scope; the existing `I18nPropertiesSyncTest.checkI18nPropertyFilesAreInSync` will mechanically validate that all 7 non-base locale files include the 3 new keys when the PR's CI runs.
