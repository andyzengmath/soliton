## Findings

### CRITICAL — `addVisit` button uses variable instead of message key (diff line 458)

`src/main/resources/templates/pets/createOrUpdateVisitForm.html`:

```html
<button class="btn btn-primary" type="submit" th:text="${addVisit}">Add Visit</button>
```

This uses `${addVisit}` (Thymeleaf variable expression) instead of `#{addVisit}` (message lookup). The model has no `addVisit` attribute (`VisitController` does not set one), so Thymeleaf will render an empty button label at runtime. The neighbouring uses in this PR all use `#{...}` correctly (e.g. `ownerDetails.html` line 351 has `th:text="#{addVisit}"`), so this is a typo, not a design choice.

**Fix**: change `${addVisit}` -> `#{addVisit}`. The translation key already exists in all 8 locale files.

### HIGH — Trailing-space inconsistency in `new=` value across locales

Only `messages.properties` (English) has a trailing space: `new=New `. The 7 translated locales (de/es/fa/ko/pt/ru/tr) all set `new=Neu`, `new=Nuevo`, `new=새로운`, etc., with no trailing space.

Templates render `new` adjacent to a sibling element:

```html
<th:block th:if="${pet['new']}" th:text="#{new}">New </th:block>
<span th:text="#{pet}">Pet</span>
```

Thymeleaf does not auto-insert whitespace between adjacent elements that have been replaced by `th:text`. The trailing space in English is therefore load-bearing — it produces "New Pet" rather than "NewPet". The translated locales will render "NeuHaustier", "Nuevo­Mascota", "새로운반려동물", etc.

**Fix options** (pick one, apply consistently):
1. Add a trailing space to all 7 translated `new=` values; OR
2. Remove the trailing space from English and put the separator in the template (e.g. wrap in a `<span>` that has CSS margin or use `&#160;` between blocks).

Option 2 is cleaner because property values with significant trailing whitespace are an established source of i18n bugs — the new `I18nPropertiesSyncTest` does not catch this and a future translator update would silently re-introduce the issue.

### IMPROVEMENT — `I18nPropertiesSyncTest` only checks key presence, not non-empty values

`checkI18nPropertyFilesAreInSync` reports a locale missing a base key, but a translator can still ship `key=` (empty value) or `key=Key` (untranslated English) and pass the test. Optional: extend the test to flag empty values. Not blocking.

### IMPROVEMENT — `messages_en.properties` is intentionally empty but not modified by this PR

The fallback file documents itself as empty. Just noting this for the reviewer record — no action needed.

### Translations themselves: not evaluated

Soliton does not evaluate translation accuracy. The 7 locales × 15 keys (105 entries) of de/es/fa/ko/pt/ru/tr translations are accepted as-submitted.
