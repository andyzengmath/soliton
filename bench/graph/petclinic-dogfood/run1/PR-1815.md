## Findings

### NITPICK — Mutability semantic change is real but unexploited (conf 78, suppressed by threshold)

`org.assertj.core.util.Lists.newArrayList(...)` returns a `java.util.ArrayList` — mutable, supports `.add/.remove/.set/.clear`. `java.util.List.of(...)` returns an immutable `ListN` (Java 9+) that throws `UnsupportedOperationException` on any mutation. This is the legitimately load-bearing semantic difference of the refactor.

I checked every call site for downstream mutation:

1. **`OwnerControllerTests:95`** — `new PageImpl<>(List.of(george))` is wrapped immediately. The `BeforeEach` block does `george.getPet("Max").getVisits().add(visit)` two lines later (line 100), but that mutates the `Visit` collection on the *Pet* (still mutable) — not the outer list. Safe.
2. **`OwnerControllerTests:144, 151, 160`** — each `Page<Owner> tasks = new PageImpl<>(List.of(...))` is consumed only by `when(...).thenReturn(tasks)` and a subsequent `mockMvc.perform(...)` GET. Spring's `OwnerController.processFindForm` reads the page via `Page.getContent()` / iteration only — no mutation. Safe.
3. **`PetControllerTests:72`** — `given(this.owners.findPetTypes()).willReturn(List.of(cat))`. The list is consumed by `PetController.populatePetTypes(...)` which iterates for the form select widget. No mutation. Safe.

Spring Data's `PageImpl(List<T> content)` constructor stores the reference directly; it does not defensively copy and it does not mutate. `MockMvc.perform(...)` does not touch the underlying list.

Verdict: the immutability change is a **real semantic delta** but **unexploited** in the touched call sites. The refactor is safe **as-shipped**. A future test author who adds `tasks.getContent().add(...)` after one of these lines would now hit `UnsupportedOperationException` — that's a future-PR concern, not a finding for this PR.

This finding is below the synthesizer's confidence-85 threshold (and is also not actionable in this diff) and is therefore *suppressed* from the actionable section. Recorded here for audit completeness.

### Hallucination check: clean

`java.util.List.of()` (no-arg) and `List.of(E)` / `List.of(E, E)` (varargs/fixed-arity) all exist in JDK 9+. PetClinic's `pom.xml` targets Java 17. All five usages match valid signatures:
- `List.of()` — empty immutable list. JDK 9+. OK.
- `List.of(george)` — single-element. OK.
- `List.of(george(), new Owner())` — two-element. OK.
- `List.of(cat)` — single-element. OK.

`java.util.List` is imported correctly in both files. The `org.assertj.core.util.Lists` import is removed correctly in both files (no orphan import remains; both files were re-scanned).
