## Findings

### Improvement-1 ‚Äî Maven/Gradle parity gap on `spring-boot-starter-restclient`

**File:** `pom.xml` lines ~125-129 (added) vs `build.gradle` line 50 (unchanged)
**Severity:** Improvement
**Source:** cross-file-impact

The PR adds a new test-scope dependency in Maven:
```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-restclient</artifactId>
  <scope>test</scope>
</dependency>
```
but `build.gradle` only carries `spring-boot-starter-restclient-test`, which transitively brings the `RestClient` core API in most setups but is not guaranteed to mirror Maven's exact classpath. PetClinic deliberately publishes both build systems; any future test that imports `org.springframework.web.client.RestClient` directly will compile in Maven and fail in Gradle.

**Suggestion:** Mirror Maven by adding `testImplementation 'org.springframework.boot:spring-boot-starter-restclient'` to `build.gradle`, or document why the asymmetry is intentional.

### Improvement-2 ‚Äî Verify `cyclonedxDirectBom` task exists at plugin version 3.2.0

**File:** `build.gradle` lines 65-68 (added)
**Severity:** Improvement
**Source:** correctness + hallucination

```gradle
tasks.named("cyclonedxDirectBom").configure {
  dependsOn("compileJava")
  skipConfigs = [".*[Tt]est.*"]
}
```

The cyclonedx-gradle-plugin upgraded `3.0.2 ‚Üí 3.2.0` and a new task `cyclonedxDirectBom` is referenced. The plugin's v3.x line did introduce direct/transitive BOM split, and `skipConfigs` is documented on the `CycloneDxTask`. The configuration is plausible.

Two concerns to verify locally:

1. **Task lookup will fail at configuration time** if the plugin name differs in 3.2.0 (e.g. `cyclonedxDirectBom` vs `cyclonedxDirectBomGenerate`). `tasks.named(...)` is eager ‚Äî this fails the build, not lazily. Run `./gradlew tasks --group=cyclonedx` once on this branch to confirm.
2. **`dependsOn("compileJava")` is redundant in most Spring Boot Gradle setups** ‚Äî the cyclonedx tasks usually attach themselves to `assemble` / consume runtime classpath which already triggers `compileJava`. Adding the explicit edge is harmless but indicates a workaround for a real ordering bug; worth a code comment for the next maintainer. Won't break parallel ordering ‚Äî Gradle's task graph deduplicates dependencies and the dependency is acyclic.

### Improvement-3 ‚Äî Mixed `if`-statement brace style in changed file

**File:** `src/main/java/org/springframework/samples/petclinic/owner/PetController.java` lines 109-111
**Severity:** Improvement (style consistency)
**Source:** correctness

The PR adds braces to one `if`-without-braces in `processCreationForm`:
```java
if (StringUtils.hasText(pet.getName()) && pet.isNew() && owner.getPet(pet.getName(), true) != null) {
    result.rejectValue("name", "duplicate", "already exists");
}
```
but PetClinic's checkstyle config (`spring-javaformat`) generally allows brace-less single-statement `if` blocks, and the surrounding code (e.g. line 113-117) keeps that style. Either enforce a project-wide brace policy in checkstyle or revert this single change to match neighboring style. Otherwise this is the only braces-everywhere `if` in the file.
